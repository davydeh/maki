# maki v0.2 Workspace Flow Design

## Goal

Add the next workspace lifecycle to maki:

- open a project folder from a native picker
- remember recent projects and the last opened project
- reopen the last project on launch by default
- support multiple windows for different projects
- focus an existing window instead of opening a duplicate for the same folder
- replace the missing-config error state with an in-app config wizard

This design is intentionally scoped to the project-opening and config-creation flow. It does not redesign the existing terminal workspace UI.

## Product Decisions

- Keep the current dark maki visual language and palette
- Show recent projects by project name first, with the full path as secondary muted metadata
- Reopen the last project automatically on launch
- Allow multiple windows at the same time for different project folders
- If a folder is already open in another window, focus that window instead of opening a duplicate
- Show the config wizard only when the selected folder has no `maki.yaml`
- Keep Python detection limited to app-server-oriented suggestions for this pass

## Recommended Architecture

Use a frontend-led flow with a thin Rust services layer.

React owns the visible app state machine and all wizard interactions. Rust owns native capabilities and persistence: folder dialogs, filesystem inspection, app-state storage, and window routing. This fits the current codebase because the existing workspace UI is already frontend-driven, while the new requirements depend on native Tauri features that should stay in Rust.

## App States

The root app flow should move between four top-level states:

1. `booting`
2. `project-ready`
3. `project-missing-config`
4. `project-invalid`

### `booting`

The app loads persisted state, resolves the last project, and decides which screen to show.

### `project-ready`

The selected project contains a valid `maki.yaml`. The app loads config, initializes tabs, and shows the existing terminal workspace.

### `project-missing-config`

The selected project folder exists but does not contain `maki.yaml`. The app keeps the normal dark shell and replaces the old error view with the config wizard.

### `project-invalid`

The last project cannot be opened because it was moved, deleted, or is unreadable. The app falls back to a recent-project picker with an `Open Folder...` action.

## Startup Behavior

On launch, maki should:

1. Load persisted app state
2. Read the last-opened project, if one exists
3. Inspect that folder
4. If the folder is valid and not already open in another window, restore it immediately
5. If the folder is valid but already open in another window, focus the existing window and avoid duplication
6. If the folder exists but has no `maki.yaml`, open the wizard for that folder
7. If the folder is invalid or unreadable, fall back to the recent-project picker

The picker should also be reachable from the running app so the user can switch projects without quitting.

## Window Model

Each project window is associated with exactly one project folder.

- Different folders may be open in different windows at the same time
- The same folder may not be opened in more than one window
- Opening a folder that already has a live window should focus that window
- Opening a different folder from the picker should create or reuse a window according to the routing rules

Window routing should live in Rust because it depends on Tauri window management and should not be reconstructed in frontend-only state.

## Frontend Components

The current [`App.tsx`](/Users/pcil/projects/maki/src/App.tsx) should stop owning every concern directly. The v0.2 flow should introduce a small app-state layer plus focused screens.

### Root Session State

A `useWorkspaceSession` hook or equivalent controller should own:

- startup state
- selected project folder
- persisted recent-project data
- config loading
- transitions between picker, wizard, and workspace
- keyboard shortcuts for workspace-level actions like `Cmd+O`

### Project Picker View

A dedicated picker view should render:

- recent projects in most-recent-first order
- primary label as project name only
- secondary line with the full folder path
- `Open Folder...`
- a clear recovery path when the last project is missing or unreadable

This is a functional fallback state, not a redesign of the app shell.

### Config Wizard View

A dedicated wizard view should render the missing-config flow in the current dark style. It should keep the interaction inline, in the same window, instead of bouncing the user to CLI setup.

### Existing Workspace View

The existing tab bar, terminal area, and status bar remain the main workspace UI once config exists. The goal is to feed them from a better project/session lifecycle, not replace them.

## Rust Service Surface

Rust should expose a focused set of commands for native behavior.

- `open_folder_dialog`
  - open Tauri's native folder picker
- `load_app_state`
  - read persisted recents, last project, and related workspace metadata
- `save_app_state`
  - persist recents and last project updates
- `inspect_project_folder`
  - determine whether the folder exists, whether `maki.yaml` exists, and which stack signals are present
- `generate_config_preview`
  - turn wizard selections into YAML text
- `save_config`
  - write `maki.yaml` into the selected folder
- `open_project_window`
  - open or focus a project window according to folder-based deduplication rules

The commands should be narrow and composable. The frontend should assemble UX behavior from them instead of relying on a single oversized boot payload.

## Core Data Flow

1. The app boots and loads persisted app state from Rust.
2. The frontend resolves the initial folder to inspect, usually the last opened project.
3. Rust inspects the folder and returns:
   - project name
   - canonical path
   - whether the folder exists
   - whether `maki.yaml` exists
   - detection signals for Node, Laravel, and Python
4. If config exists, the frontend loads `maki.yaml` and initializes tabs.
5. If config is missing, the frontend enters the wizard with the detection result preloaded.
6. If the user selects `Open Folder...`, Rust returns a folder path from the native dialog.
7. Rust updates recent-project persistence and applies window routing.
8. If the wizard saves successfully, Rust writes `maki.yaml`; the frontend reloads the project and launches tabs immediately.

## Wizard Scope

The wizard is deliberately narrow for v0.2. It should solve the first-run path without becoming a full project-inspection engine.

### Step 1: Detect Stack

Supported detections for this pass:

- **Node**
  - detect `package.json`
  - suggest common scripts such as `dev`, `start`, `serve`, and `watch` when present
- **Laravel**
  - detect `artisan` and/or `composer.json`
  - suggest `php artisan serve`
  - suggest `php artisan queue:work` when appropriate
  - suggest `npm run dev` when frontend tooling is present
- **Python**
  - detect `pyproject.toml`, `requirements.txt`, `requirements-dev.txt`, `manage.py`, or common app entrypoints
  - prefer app-server suggestions only for this pass, such as Django, FastAPI, or Flask
  - fall back to generic editable commands when confidence is low

### Step 2: Suggested Commands

Show a checklist of suggested commands with editable fields:

- enabled checkbox
- display name
- command string

If nothing is detected, the wizard should still work with an empty list and an obvious `Add command` action.

### Step 3: Autostart Defaults

For each enabled command, let the user choose whether it should autostart. The default may come from the detector, but the user must be able to change it before saving.

### Step 4: YAML Preview

Show the exact generated `maki.yaml` text before saving so the user understands what will be written.

### Step 5: Save And Launch

Save `maki.yaml` into the project folder and transition straight into the normal workspace by loading the new config and opening tabs.

## Persistence Model

Persist an app-state document outside the project folder that contains:

- recent projects
- last opened project
- enough metadata to render the picker quickly, such as display name and canonical path

Recents should be updated whenever a project is successfully selected or restored. The most recently opened project should always be first.

## Error Handling

- If the last project is missing, fall back to the picker and keep recents intact
- If the folder is unreadable, show an inline error and allow re-picking a folder
- If config parsing fails after save, keep the user in a recoverable state and show the error clearly
- If saving `maki.yaml` fails, keep wizard state in memory and do not discard edits
- If no stack is detected, keep the wizard usable with manual command entry
- If a duplicate window open is requested, focus the existing window instead of creating another

## Testing Strategy

The implementation plan should include:

- Rust tests for app-state persistence, folder inspection, and config generation
- frontend tests for the root session state machine and wizard transitions
- smoke verification for `Cmd+O`, recent-project restore, and duplicate-window focusing
- manual terminal verification after the flow lands:
  - typing works
  - ANSI colors render correctly
  - resize remains smooth

## Explicitly Out Of Scope

- redesigning the current workspace visual style
- multiple windows for the same project folder
- deep framework inference beyond Node, Laravel, and Python
- importing `.env` into the wizard UI
- manual recent-project metadata editing
- a separate launcher-only home window
- GitHub repo creation as part of this design

## Success Criteria

The feature is successful when:

- maki restores the last project on launch when possible
- users can open another folder with `Cmd+O` and the native picker
- duplicate project windows are prevented by focusing an existing window
- folders without `maki.yaml` go directly into a usable in-app wizard
- saving from the wizard writes `maki.yaml` and launches tabs immediately
- the existing dark terminal workspace remains visually consistent
