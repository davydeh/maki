## What Failed

The multi-window project-routing flow is still unreliable in native end-to-end use.

Observed symptom:

- Opening another project could still produce a second macOS app/window flow that did not consistently land on the expected new picker/wizard/workspace state
- At least once, the user saw the old legacy missing-config screen again in the newly opened app/window even though the new UI also existed elsewhere
- macOS Dock behavior was also not where the product should end up:
  - multiple Dock icons/app instances feeling wrong
  - desired behavior is closer to Chrome: one Dock icon and Cmd+` window cycling

## What Was Tried

### 1. Initial multi-window routing

Implemented a Rust-side project window registry keyed by canonical project path.

Intent:

- if project already open, focus that window
- if different project, open a new `project-*` window
- keep frontend session state local to each window

### 2. Window binding and restore fixes

Several follow-up fixes tightened when the frontend binds the current window to a project and when restore uses local state versus native window routing.

### 3. Capability fix for project windows

Native dev verification exposed a Tauri capability problem:

- `event.listen not allowed on window "project-1"... allowed on [windows: "main", URL: local]`

Fix applied:

- widened `src-tauri/capabilities/default.json` from `main` only to `main` plus `project-*`

Commit:

- `dbf838b`

This removed one concrete runtime blocker, but it did not fully resolve the overall multi-window behavior.

### 4. Boot precedence fix for stale persisted state

Best concrete root-cause finding during debugging:

- a newly created `project-*` window could start up before persisted `last_project_path` had caught up
- boot logic was consulting persisted app state too early
- that meant the new window could bootstrap against stale project state rather than its own bound project

Fix applied:

- boot now prefers `get_current_project_window` before falling back to persisted `last_project_path`

Commit:

- `94ba0fb`

This fix is test-covered and likely correct, but native end-to-end confidence is still not high enough to keep pushing the multi-window flow in the current session.

## Best Guess About What Is Broken

The most likely problem is that the multi-window architecture is still mixing two authorities during startup:

- persisted global app state (`last_project_path`, recents)
- live per-window project binding (`get_current_project_window`)

That creates fragile timing and ownership problems during:

- new window creation
- restore
- switching from one already-open project to another
- configless-project wizard entry

Even after the boot-precedence fix, the product behavior still does not feel settled enough at the macOS app/window level. There may be another bootstrap or native-window lifecycle gap that only shows up in real app usage and not in the unit tests.

There is also a separate product-level concern:

- even if the routing were made reliable, the current multi-window UX is not yet aligned with the desired macOS behavior

## Recommended Next Session

Do not resume by adding more small fixes to the current multi-window path blindly.

Recommended first decision:

1. Either intentionally simplify to single-window v0.2
2. Or pause and redesign the multi-window architecture around a single source of truth for startup and routing

My recommendation is to simplify to single-window first.

That would mean:

- `Open Folder` replaces the current project in-place
- recents open in the current window
- last-project restore stays in the current window
- picker and wizard stay
- Rust project-window routing becomes unused for now

Reason:

- most of the valuable v0.2 work is already in place
- the unstable part is the multi-window routing layer
- this gives a shippable flow faster and leaves room to revisit macOS-native multi-window behavior later
